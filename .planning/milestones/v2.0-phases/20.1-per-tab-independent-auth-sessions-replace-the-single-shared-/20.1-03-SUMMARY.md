# Phase 20.1 Plan 03: Fetch-Wrapper Transport + Silent Refresh + Entry/Sign-Out UI Summary

**Status: COMPLETE.** All 4 tasks and both blocking-human checkpoints (Task 1.5, Task 4) are done. Two real bugs surfaced during Task 4's live multi-tab verification and were fixed and re-verified before sign-off.

## Status

### Task 1 — tab-refresh Route Handler
Complete, unchanged from the original commit (`3749c09`). `app/api/auth/tab-refresh/route.ts`: `POST` reads `{ refreshToken }`, calls `verifyTabToken`, requires `typ === 'refresh'`, mints a new access token via `mintTabAccessToken`, returns `{ accessToken, expiresAt }`. Never calls `verifySession()`. `tests/app/api/tab-refresh.test.ts`: 5/5 passing (valid refresh → 200; invalid/expired → 401; `typ='access'` presented as refresh → 401; malformed body → 400; missing field → 400).

### Task 1.5 — [BLOCKING] fetch-wrapper survives soft navigation
Confirmed. Task 2's implementation and its in-code comment record this was verified against this exact Next 16.2.9 + React 19.2.4 install before Tasks 2-3 were built on top of it, per D-20.1-03-A. (This confirmation predates this session — the code for Tasks 2-3 already existed on disk, matching the plan's spec, when this session picked the plan back up.)

### Task 2 — TabSessionProvider + root-layout mount
Complete. `app/_components/tab-session-provider.tsx` is `'use client'`, gates its `window.fetch` override on `sessionStorage.getItem('tabAccessToken')`, and is mounted in `app/layout.tsx` wrapping `{children}` inside `<body>`. **This session found and fixed a real bug in this task's design**: the provider's mount-once `useEffect` never re-checks sessionStorage, so a token minted mid-session (via the "sign in as a different user" flow, which never remounts the provider) was silently missed — that first navigation went out with no Authorization header and resolved to the shared cookie's identity instead of the new one. Fixed by adding a `TAB_SESSION_ACTIVATE_EVENT` window event: `new-session-form.tsx` now dispatches it synchronously, before `router.push`, and the provider listens for it in addition to its mount check.

### Task 3 — entry link + per-tab sign-out control
Complete. `sign-in-form.tsx` has the `<a href="/sign-in?newSession=1">Sign in as a different user</a>` link. `tab-sign-out-button.tsx` clears exactly the 3 sessionStorage keys and hard-navigates to `/sign-in`, with zero references to `signoutAction`/`signOut(` (grep-verified: 0). `app/(app)/layout.tsx` renders `<TabSignOutButton />` beside `<SignOutButton />`. **This session found and fixed two things here**:
- `tab-sign-out-button.tsx` originally called `setState` synchronously inside a `useEffect` body (`npm run lint` failed with `react-hooks/set-state-in-effect`). Rewritten to use `useSyncExternalStore`, matching the existing pattern in `theme-toggle.tsx`.
- Its doc comment contained the literal substring `signoutAction` (in prose, not code), which the acceptance criteria's own grep check would have flagged. Reworded.

### Task 4 — [BLOCKING] end-to-end token transport verification
**Approved.** Verified live against the running dev server using two real seeded QA accounts (`qa.factory@trtarredo.demo` / factory_pm via the shared cookie, `qa.ops@trtarredo.demo` / operations via the per-tab flow — `qa.site@trtarredo.demo` referenced in prior QA notes no longer exists in the DB and was substituted; its stale password was also reset), driven via automated browser tooling (agent-browser) rather than manual DevTools clicking. All 7 steps passed:
1. Tab A signs in normally as qa.factory (shared cookie) → dashboard loads as qa.factory. ✓
2. "Sign in as a different user" → credentials form appears, no redirect (D-03 carve-out). ✓
3. Signs in as qa.ops in the same tab → navigates to `/admin/dashboard`, renders as qa.ops (operations), not the original qa.factory. ✓ *(This step failed on the first two attempts — see bugs below — and passed after fixes.)*
4. DevTools-equivalent check (`sessionStorage` via `eval`) confirms `tabAccessToken`/`tabRefreshToken`/`tabTokenExpiresAt` are set in `sessionStorage`, not `localStorage`. ✓
5. A genuinely separate Tab B (new tab, not duplicated) to the dashboard URL still shows the ORIGINAL qa.factory user — unaffected by Tab A (D-01). ✓
6. Client-side navigation within Tab A retains the qa.ops identity — confirmed both positively (Authorization header present on every soft-nav fetch) and negatively (a 403 on `/factory-pm/projects`, a factory_pm-only route, proves the operations role is genuinely enforced server-side via the header, not cosmetic). ✓
7. "Sign out (this tab)" in Tab A clears sessionStorage and hard-navigates to `/sign-in`, which immediately falls through to Tab A's still-valid shared cookie (qa.factory) — expected, since that cookie was never touched. Tab B, checked immediately after, is completely unaffected throughout (D-07). ✓

**Two real bugs were found and fixed during this checkpoint** (both committed in `2ff12d0`, after the Task 2/3 lint fix in `65c2cb0`):
1. **TabSessionProvider's mount-once check** (described under Task 2 above) — the root cause of step 3 failing on the first two attempts (identity silently fell back to the shared cookie instead of the new user).
2. **`app/(app)/layout.tsx` read identity via `auth()` directly instead of `verifySession()`.** Even after fixing bug 1, the *page* content correctly showed the new user (qa.ops/admin dashboard tiles) but the *sidebar* kept showing the OLD user's (qa.factory/factory_pm) nav shape — because the shared layout never looked at the Authorization header at all. This affected the sidebar, header name/avatar/role label, and every value threaded from the layout into `HeaderProjectSwitcher`, `MyWorkProvider`, and `PendingStepGate`. Fixed by switching the layout to `verifySession()` and sourcing `name` from a DB lookup (per-tab tokens don't carry a name).
