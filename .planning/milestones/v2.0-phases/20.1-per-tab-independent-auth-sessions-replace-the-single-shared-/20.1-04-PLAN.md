<objective>
Close the one gap RESEARCH.md's own Pattern 3 / Pitfall 2 flagged but its architecture summary understated: Server Actions cannot receive a client-attached `Authorization` header (confirmed via `node_modules` behavior and two independent Next.js GitHub discussions) — the header-first DAL adapter built in Wave 0 transparently covers Server Components and Route Handlers, but a Server Action invoked from a per-tab-session tab has no header to read at all, and will silently fall through to whichever user's shared cookie happens to be in that browser. That silent-fallback behavior is exactly the "shows the wrong user" bug class this entire phase exists to fix, now surfacing at the one layer the header mechanism cannot reach.

This plan: (1) adds a DAL-consistent, explicitly-token-scoped sibling to `verifySession()` for Server Actions to opt into, reusing the exact same crypto/fail-closed logic (no duplicated verification code — the DAL stays the sole choke point per this repo's established convention); (2) proves the full bound-argument pattern end-to-end on one real, representative, low-risk Server Action (`updateProfileAction`); and (3) makes an explicit, documented scope decision — given ~57 total DAL call sites exist across the app and most are Server Actions invoked from plain server-rendered `<form action={fn}>` markup (no client wrapper today), migrating every one of them to the bound-argument pattern is a repo-wide refactor far beyond this phase's reasonable scope. This plan delivers the reusable mechanism and one proof-of-concept, and explicitly defers the remaining rollout — mirroring this project's own established precedent (Phase 19-04's ROLE-02 finding: "left Partial rather than force-flipped to complete," a genuine gap recorded, not hidden) rather than silently claiming full D-06 coverage that was never actually built.

Purpose: gives the phase a real, working example of the harder half of D-06 (Server Actions), and an honest, load-bearing statement of what remains — protecting whoever picks this project back up from believing "sign in as a different user" makes EVERY interaction independent when today it only makes navigation/viewing independent plus this one action.
Output: `lib/dal.ts` extended with `verifySessionForAction`, a small client hook (`lib/use-tab-token.ts`), `actions/profile.ts` migrated to the bound-argument pattern, `app/(app)/profile/page.tsx` + new `app/_components/profile-form.tsx` wired end-to-end, full regression suite green, and a final phase-level manual sign-off checkpoint.
</objective>

<decision>
D-20.1-04-A (scope boundary, load-bearing): This phase migrates exactly ONE Server Action (`updateProfileAction`) to the bound-argument pattern as a proof-of-concept and reusable template. The other ~56 `verifySession()`/`requireRole()`/`requireAdmin()`/`requireOwnerOrAdmin()` call sites (the large majority being Server Actions across `checklists.ts`, `workflow.ts`, `workflow-graph.ts`, `readiness.ts`, `processes.ts`, etc.) are explicitly NOT migrated this phase. A per-tab-session tab calling one of those unmigrated actions today will act as whatever the SHARED COOKIE currently holds in that browser — not an error, not a crash, just silently the "other" identity. This is recorded here as a known, accepted, scoped limitation (not fixed, not hidden) — full rollout is a natural fast-follow quick-task once the pattern is proven, sized at roughly one client-wrapper + one signature change per remaining action.

D-20.1-04-B (no duplicated crypto logic): `verifySessionForAction(explicitToken?: string)` must call the SAME internal token-decode/typ-check/fail-closed logic that Wave 0's `verifySession()` header path uses — extract that shared body into a private, un-exported helper in `lib/dal.ts` (e.g. `resolveTabIdentity(rawToken: string)`) that both `verifySession()`'s header branch and the new `verifySessionForAction()` call. Two independent implementations of "decode + check typ + fail closed" would be exactly the kind of divergent-copy risk this repo's own `verifyCredentials()` extraction (Wave 0, Task 1) was designed to avoid.

D-20.1-04-C (why `updateProfileAction` specifically): it is single-purpose, touches only the caller's own row (no cross-user authorization complexity to reason about), already `FormData`-shaped (mirrors the sign-in action's calling convention), and has a low blast radius if the migration has a bug (worst case: a profile edit fails or targets the wrong-but-still-authenticated user, not a data leak across roles or a workflow-integrity issue).
</decision>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client-bound `tabToken` argument -> `updateProfileAction` | The token travels as an ordinary Server Action argument (Next.js serializes it like any other parameter) — same cryptographic trust as the header path, just a different transport |
| Unmigrated Server Actions (56 remaining) -> shared cookie | Explicitly scoped OUT of this phase; a per-tab tab calling one of these still resolves via whatever cookie identity is present in that browser |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-20.1-12 | Elevation of Privilege | A forged/tampered `tabToken` argument passed directly to `updateProfileAction` (bypassing the client UI) | mitigate | `verifySessionForAction` runs the identical cryptographic `resolveTabIdentity`/`verifyTabToken` check as the header path — a forged argument fails to decode exactly like a forged header would; Server Action arguments are still server-executed code, not client-trusted state |
| T-20.1-13 | Spoofing | An unmigrated Server Action silently acting as the wrong (shared-cookie) user when called from a per-tab tab | accept | Explicitly documented per D-20.1-04-A and surfaced live in the Task 4 checkpoint (step 9) rather than hidden — full mitigation (migrating all 56 remaining call sites) is out of this phase's scope and filed as a fast-follow; this phase's actual security improvement is that Server Components/Route Handlers (the majority of the read/navigation surface) are fully covered, and the one migrated mutation proves the pattern works when applied |
| T-20.1-14 | Repudiation | Two divergent token-verification implementations drifting apart over time (header path vs. bound-argument path) | mitigate | D-20.1-04-B: both paths call the same private `resolveTabIdentity` helper — a future fix/change to verification logic cannot be applied to only one path by accident |
</threat_model>
