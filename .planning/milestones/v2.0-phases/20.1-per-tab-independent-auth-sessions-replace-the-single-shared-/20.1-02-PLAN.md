<objective>
Build the "sign in as a different user" entry flow: a new Server Action that verifies credentials and mints per-tab tokens WITHOUT touching the shared cookie (so it can never disturb other open tabs), and the sign-in page carve-out (D-03) that lets this flow reach the credentials form even when a shared-cookie session already exists in that tab. This is the mechanism a user actually triggers â€” Wave 0's DAL/token primitives (20.1-01) exist to serve this flow, and Wave 2 (20.1-03) will build the transport (fetch-wrapper header injection, refresh) that makes the minted tokens actually authenticate subsequent requests.

Purpose: closes D-01/D-02/D-03/D-04 â€” the trigger UX and the credential-verification/token-minting half of the feature. Token *transport* (getting the header onto outgoing requests) and the visible entry-point link/button placement are Wave 2's job (20.1-03), kept separate so this plan's tests stay focused on the sign-in action and page-carve-out logic in isolation.
Output: `actions/tab-auth.ts` (tabSigninAction), `app/(auth)/sign-in/new-session-form.tsx` (client form), `app/(auth)/sign-in/page.tsx` extended with the `newSession` searchParam carve-out, and a small pure-function extraction (`lib/auth/sign-in-redirect.ts`) so the redirect-vs-render decision is unit-testable without a Server Component rendering harness (this repo has no existing RSC page-test pattern â€” see Decision below).
</objective>

<decision>
D-20.1-02-A (testable redirect logic, since this repo has no RSC page-test pattern): `tests/actions/auth.test.ts` only covers Server Actions, never Server Component pages â€” confirmed during Wave 0 research, no existing precedent to extend. Rather than introduce a new, heavier RSC-rendering test harness for one boolean decision, extract the D-01/D-03 redirect-vs-render logic into a plain, framework-free function `shouldRedirectFromSignIn(hasSession: boolean, newSession: boolean): boolean` in `lib/auth/sign-in-redirect.ts`, unit-test that directly, and have `page.tsx` call it. This keeps `page.tsx` itself a thin 6-line wrapper (matching its current minimalism) while making the actual decision fully covered by fast unit tests.

D-20.1-02-B (no redirect() inside tabSigninAction â€” mandatory per RESEARCH.md Pattern 3/5 and Pitfall 3): Server Components render before any client `useEffect` runs, so a server-side `redirect()` here would discard the freshly-minted tokens with nowhere for the client to read them from. `tabSigninAction` MUST `return { accessToken, refreshToken, expiresAt }` (or `{ message }` on failure) and let the calling Client Component drive `sessionStorage` + `router.push()` itself.

D-20.1-02-C (never call next-auth's signIn() from this path â€” mandatory per RESEARCH.md Anti-Patterns): `signIn()` sets the shared cookie directly, which would silently sign in every OTHER tab as this new user â€” reproducing the exact bug this phase exists to fix. `tabSigninAction` calls `verifyCredentials()` directly, never `signIn()`.
</decision>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| `tabSigninAction` form input -> `verifyCredentials()` | Same credential-verification trust boundary as the existing `signinAction` â€” untrusted email/password from the client |
| `new-session-form.tsx` client state -> `sessionStorage` | Tokens returned from a Server Action land in ordinary React state before being written to `sessionStorage` â€” same-origin JS can read them (accepted per D-05, this phase's locked trade-off) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-20.1-05 | Spoofing | `tabSigninAction` credential brute-force | accept | Same exposure as the existing `signinAction` (no new attack surface) â€” rate-limiting is out of scope for both, unchanged from today's posture |
| T-20.1-06 | Tampering | `new-session-form.tsx` writing tokens to `sessionStorage` before validating them | mitigate | Tokens are opaque signed JWEs freshly returned from a trusted Server Action response in the same request/response cycle â€” nothing to validate client-side beyond presence, since forging a valid JWE client-side is cryptographically infeasible without `AUTH_SECRET` |
| T-20.1-07 | Elevation of Privilege | A malicious `newSession` searchParam value bypassing the D-03 carve-out unexpectedly | accept | `shouldRedirectFromSignIn` treats ANY truthy `newSession` value as "show the form" â€” this is intentional (the carve-out is meant to be reachable by a simple link, not secret), and reaching the sign-in FORM is not privileged; no session is granted until valid credentials are submitted through the normal `tabSigninAction` verification path |
</threat_model>
