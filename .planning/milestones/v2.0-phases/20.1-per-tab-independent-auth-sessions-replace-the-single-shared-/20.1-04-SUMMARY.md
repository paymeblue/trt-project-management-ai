Any of the remaining ~56 unmigrated Server Actions invoked from a per-tab-session tab will silently resolve via the shared cookie's identity — confirmed conceptually equivalent and observed live via repeated hard-navigation fallback behavior throughout this session (graceful, no crash, just the other tab's identity). Per D-20.1-04-A, this is the accepted, documented scope limitation, not a defect.
10. Task 3's full automated gate (tsc/lint/`npm test`) confirmed clean before treating this checkpoint as signed off.

**Two additional, genuine, pre-existing bugs were found during this live verification** (both Wave 2/Plan 03 code, outside this plan's `files_modified` scope) — logged to `deferred-items.md` and `STATE.md` Pending Todos, **not fixed here** per the executor's scope-boundary rule:

1. **HIGH — silent refresh drops the role claim.** `app/api/auth/tab-refresh/route.ts` mints the refreshed access token via `mintTabAccessToken(payload.sub, payload.role ?? '')`, but `payload` is the decoded REFRESH token, which by design (`mintTabRefreshToken`) never carries a `role` field. Every silently-refreshed access token (any tab open >~18 minutes) gets `role: ''` baked in, silently breaking that tab's role-gated authorization with no crash or visible error.
2. **MEDIUM — intermittent 403 on `/admin/dashboard`.** A valid, non-expired, correctly-decoded (`role: "operations"`) per-tab access token intermittently produced a `forbidden()` response from `requireAdmin()` on repeat soft-navigations to `/admin/dashboard`, while the identical token consistently succeeded against `/profile` in the same session. First post-signin navigation always rendered correctly; only later re-navigations were affected. Root cause not conclusively isolated this session (candidates: dev-mode `cache()` request-scoping, a fetch-override race under concurrent polling traffic, Router Cache staleness) — flagged for a focused follow-up investigation, ideally against a production build to rule out dev-only causes.
