<objective>
Complete `ROLE-05`: position is NOT collected at account creation, and a user sets their own position afterward via a self-service profile flow constrained to the position enum's valid values. Now that `POSITION_VALUES` (19-01) is a DB-enforced enum, replace the profile page's free-text `<input list=datalist>` with a real `<select>`, add matching server-side validation, strip the position field from the admin create-user flow, and convert the Configurator's `requiredPosition` control from a known-buttons+free-text-custom fallback into an enum-constrained select (so an out-of-enum value can never be authored and then fail on insert into `users.position`).

Purpose: Closes the last gap between the shipped ad-hoc free-text position UI and ROLE-05's "constrained to the position enum's valid values" plus "not collected at account creation."
Output: profile select + validation, admin creation form with no position, configurator enum-backed position select.
</objective>

<decision>
D-19-03-A: The profile `<select>` offers ALL POSITION_VALUES (not a role-scoped subset). Role-scoping the offered titles is out of scope for Phase 19 — the requirement is only "constrained to the position enum's valid values." Option labels use `POSITION_LABELS[v] ?? v` so snake_case machine values render as friendly titles while verbatim super-admin display titles render as-is.
D-19-03-B: The Configurator drops the `__custom__` free-text fallback entirely. Because `users.position` is now an enum, a `requiredPosition` set to a non-enum string could never match any user and would be a silent dead gate — so the control must only offer enum values plus "Anyone with this role" (null).
</decision>
