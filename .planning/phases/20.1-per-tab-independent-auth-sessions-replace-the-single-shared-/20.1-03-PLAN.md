<objective>
Make the per-tab tokens minted in Wave 1 actually authenticate requests: install a global `fetch` wrapper (mounted once near the app root, active only for tabs holding a per-tab token) that injects the `Authorization: Bearer` header onto same-origin requests and Next's own soft-navigation RSC fetches; add a silent-refresh mechanism so a 20-minute access token doesn't strand a still-open tab; and surface the two pieces of UI the whole feature has been building toward — the "sign in as a different user" entry link and a per-tab-only sign-out control.

This is also where RESEARCH.md's one MEDIUM-confidence assumption (A1: the fetch-wrapper pattern actually works for Next's RSC soft-navigation fetches in this exact Next 16.2.9 setup — community-verified against Next 15.x, not officially documented, not yet confirmed against this codebase's version) gets tested for real, as the FIRST task in this plan, before the remaining tasks are built on top of it — per RESEARCH.md's own explicit recommendation.

Purpose: closes D-04 (transport), D-05 (refresh mitigating the XSS-exposure trade-off), D-07 (per-tab sign-out), and the D-02 discretion call on entry-point placement.
Output: `app/_components/tab-session-provider.tsx` (fetch wrapper + refresh timer, mounted in `app/layout.tsx`), `app/api/auth/tab-refresh/route.ts` (Route Handler), a visible entry link on the sign-in form, and a per-tab sign-out control.
</objective>

<decision>
D-20.1-03-A (smoke test gates the rest of the plan): RESEARCH.md flags the fetch-wrapper mechanism as MEDIUM confidence — community-verified on Next 15.x, not confirmed against this project's exact Next 16.2.9/React 19.2.4 install. Task 1 is a minimal, throwaway smoke test (a scratch Route Handler + a scratch client component performing one `router.push`) whose sole purpose is to visually/programmatically confirm the `Authorization` header survives a soft navigation. If it does NOT survive, STOP and escalate — do not proceed to Tasks 2-5 with an unproven transport mechanism. (If confirmed, the scratch files are deleted before Task 2 begins — they are not part of the shipped feature.)

D-20.1-03-B (silent refresh timing): the provider schedules a refresh call at `expiresAt - 2 minutes` (buffer against clock drift/in-flight request timing), calling `POST /api/auth/tab-refresh` with the stored refresh token; on success it overwrites `tabAccessToken`/`tabTokenExpiresAt` in `sessionStorage` and reschedules; on failure (refresh token itself expired, 8h boundary) it clears all 3 sessionStorage keys and leaves the tab to fall through to the shared-cookie path on its next request (graceful degradation, not a hard error) — matching this repo's established "keep last known state on transient failure" convention from `my-work-provider.tsx`.

D-20.1-03-C (entry-point placement, per CONTEXT.md's Claude's-discretion grant): the link lives on `sign-in-form.tsx` itself (visible to anyone on `/sign-in`, whether or not they currently have a session) rather than gated inside the authenticated dashboard header — this covers BOTH the "I'm signed out in this tab and want a fresh different-user session" case AND the "I already have a session in this tab but want to add a second identity" case (D-03's exact scenario) with one link, avoiding a second entry point to build/maintain.
</decision>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| `window.fetch` override -> outgoing same-origin requests | Client-side JS now decides what `Authorization` header (if any) accompanies every fetch from a per-tab-session tab |
| `POST /api/auth/tab-refresh` -> new access token | Anyone possessing a valid, unexpired refresh token can mint a fresh access token — no additional re-authentication required (standard refresh-token semantics) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-20.1-08 | Information Disclosure | `window.fetch` override exposing the token to any script running in that tab (XSS) | accept | Explicitly accepted per D-05 (locked decision) — internal tool, small known user base; mitigated by the 20-minute access-token lifetime, not by attempting to hide the token from JS (not achievable in a tab-scoped, non-httpOnly storage model) |
| T-20.1-09 | Denial of Service | `/api/auth/tab-refresh` called repeatedly/rapidly | accept | Same exposure class as the existing sign-in Server Action (no rate-limiting anywhere in this app today) — out of scope for this phase, consistent with the rest of the codebase's current posture |
| T-20.1-10 | Spoofing | `/api/auth/tab-refresh` accepting a tampered/forged refresh token | mitigate | `verifyTabToken` cryptographically verifies the JWE; a forged token fails to decode and the route returns 401 before minting anything |
| T-20.1-11 | Repudiation | Per-tab sign-out (`tab-sign-out-button.tsx`) accidentally signing out the shared cookie too | mitigate | Grep-enforced in Task 3's acceptance criteria: the file contains zero references to `signoutAction`/`signOut()` — a regression here is caught mechanically, not just by code review |
</threat_model>
