<objective>
Lay the foundation for per-tab independent auth sessions: extract the existing bcrypt credential-verification logic into a shared, reusable function; add server-only primitives to mint and verify short-lived, tab-scoped JWE tokens using Auth.js's own `next-auth/jwt` `encode`/`decode` (same encryption scheme as the main session cookie, no new crypto dependency); and extend `lib/dal.ts`'s `verifySession()` — the sole authoritative session-read choke point used by all ~57 existing call sites — with a header-first, cookie-fallback adapter that will let a later plan's per-tab tokens resolve to the correct identity without disturbing the default shared-cookie behavior for every other tab.

This plan produces no user-visible behavior change yet (no new sign-in UI, no token issuance flow) — it is pure infrastructure, isolated in its own Wave 0 plan because `lib/dal.ts` sits underneath every authenticated route in the app and any regression here breaks the entire product, not just this phase's feature.

Purpose: Wave 1 (per-tab sign-in action) and Wave 2 (token transport/refresh) both build directly on `lib/tab-session.ts`'s mint/verify functions and `lib/dal.ts`'s new header-first precedence — nothing downstream can be built or tested until this plan lands.
Output: `lib/auth/verify-credentials.ts`, `lib/tab-session.ts` (both new), `lib/dal.ts` and `auth.config.ts` extended in place, with unit tests proving the header-first/cookie-fallback precedence and the token round-trip.
</objective>

<decision>
D-20.1-01-A (salt isolation): Per-tab tokens use a distinct JWT `salt` (`"trt-pm.tab-session"`) from the shared session cookie's default salt (the cookie name). Same `AUTH_SECRET`, different salt — a per-tab token can never decode successfully as a shared-session cookie value or vice versa, even though both use the same underlying secret. This is defense-in-depth per RESEARCH.md Assumption A3, not a correctness requirement, but costs nothing to include.

D-20.1-01-B (token lifetimes, per CONTEXT.md's Claude's-discretion grant): 20-minute access token, 8-hour refresh token (roughly a working day) — both minted via `encode()` with an explicit `typ` claim (`"access"` | `"refresh"`) so one can never be presented as the other. Documented here per CONTEXT.md's instruction to record the discretionary choice.

D-20.1-01-C (fail-closed on invalid header): If an `Authorization: Bearer` header is present but `verifyTabToken()` returns null (expired/malformed/wrong-typ), `verifySession()` redirects to `/sign-in` — it must NOT fall through to the shared-cookie `auth()` path. Falling through would resolve to whichever user's cookie happens to be in the browser (a different tab's identity), reproducing the exact "shows previous user" bug this phase fixes at a different layer (RESEARCH.md Pitfall 1).
</decision>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Incoming `Authorization` header -> `verifySession()` | Any request can send an arbitrary Bearer value; only a token minted by this app's `AUTH_SECRET` + `TAB_SALT` may resolve to a real identity |
| `lib/tab-session.ts` -> `process.env.AUTH_SECRET` | Same secret the main session cookie already trusts — no new secret to provision or leak |
| `auth.config.ts` `authorized()` -> `proxy.ts` | Presence-only check; must never be treated as authoritative (DAL remains the sole authoritative check per this repo's established pattern) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-20.1-01 | Spoofing | Forged `Authorization: Bearer <garbage>` header | mitigate | `decode()` cryptographically verifies the JWE against `AUTH_SECRET`+`TAB_SALT`; any tampered/forged value fails to decode and `verifyTabToken` returns `null`, triggering fail-closed redirect (D-20.1-01-C), not a fallback |
| T-20.1-02 | Elevation of Privilege | Presenting a `typ: "refresh"` token where an access token is expected | mitigate | `verifySession()` explicitly checks `payload.typ === "access"`; a refresh-typed token is rejected even though it decodes successfully |
| T-20.1-03 | Tampering | A per-tab token replayed as a shared-session cookie value (or vice versa) | mitigate | Distinct `TAB_SALT` (D-20.1-01-A) means `decode()` with the cookie's default salt fails against a tab token and vice versa — cross-slot replay fails cryptographically, not just by convention |
| T-20.1-04 | Information Disclosure | `AUTH_SECRET` reuse across both token types | accept | Same secret Auth.js already trusts for the main session; reusing it (with a distinct salt) avoids introducing a second secret to provision/rotate/leak — standard practice for multi-purpose JWEs under one KMS-equivalent secret |
| T-20.1-01-SC | Tampering | npm/pip installs | mitigate | No new packages introduced (uses already-installed `next-auth`/`@auth/core` internals) — no legitimacy gate needed |
</threat_model>
