# Phase 20.1 Plan 02: Per-Tab Sign-In Entry Flow Summary

**New tabSigninAction Server Action mints per-tab JWE tokens without touching the shared session cookie, plus a `?newSession=1` carve-out on the sign-in page so a second tab can reach the credentials form even with an existing shared-cookie session active.**

## Performance

- **Duration:** ~25 min
- **Tasks:** 3
- **Files modified:** 6 (3 created source + 2 created test + 1 modified)

## Accomplishments
- `actions/tab-auth.ts`'s `tabSigninAction` verifies credentials via `verifyCredentials()` (Wave 0), mints a per-tab access token + refresh token via `mintTabAccessToken`/`mintTabRefreshToken` (Wave 0), and returns them directly — no cookie write, no `next-auth` `signIn()`, no server-side `redirect()`.
- `lib/auth/sign-in-redirect.ts`'s `shouldRedirectFromSignIn()` isolates the D-03 carve-out logic (skip the existing "redirect to /dashboard if already signed in" behavior when `?newSession=1` is present) as a pure, unit-tested function.
- `app/(auth)/sign-in/page.tsx` now calls `shouldRedirectFromSignIn()` instead of the old unconditional `if (session?.user) redirect('/dashboard')`.
- `app/(auth)/sign-in/new-session-form.tsx` is the client form for the `?newSession=1` path: submits to `tabSigninAction`, writes `tabAccessToken`/`tabRefreshToken`/`tabTokenExpiresAt` into `sessionStorage`, then `router.push('/dashboard')`.
