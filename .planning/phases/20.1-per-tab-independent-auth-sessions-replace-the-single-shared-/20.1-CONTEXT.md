# Phase 20.1: Per-Tab Independent Auth Sessions - Context

**Gathered:** 2026-07-17
**Status:** Ready for planning

<domain>
## Phase Boundary

Today, auth is a single shared `authjs.session-token` cookie per browser (NextAuth/Auth.js v5, JWT strategy, Credentials provider) — every tab of the same non-incognito browser shares it, so only one user identity can be active per browser at a time. Users currently work around this with incognito windows for concurrent multi-role testing/work.

This phase delivers: a mechanism for a user to deliberately start an independent, per-tab session for a *different* user, without disturbing other open tabs' sessions, and without requiring incognito. Root cause of the reported bug is documented in `.planning/debug/auth-single-session-multi-tab.md` (2026-07-17) — cookies are inherently one-per-browser-origin; the current sign-in page's unconditional `redirect('/dashboard')` when any session exists compounds the confusing "shows previous user" symptom.

**Out of scope:** Changing the default single-user, single-tab experience. A normal new tab (typed URL, opened link, browser "duplicate tab") still inherits the existing shared session for convenience — this phase does not make every tab logged-out-by-default.

</domain>

<decisions>
## Implementation Decisions

### Trigger / UX model
- **D-01:** A new tab opened normally (typed URL, clicked link, duplicate tab) still inherits the current shared session — no change to today's convenience for the common single-user case.
- **D-02:** A specific, explicit action — e.g. a "Sign in as a different user" link/button reachable from the sign-in page and/or dashboard — starts a fresh, independent per-tab session scoped to that tab only, without touching any other open tab's session.
- **D-03:** The current sign-in page behavior of unconditionally `redirect('/dashboard')` when *any* session exists must change for the "sign in as different user" entry point specifically — that path must always reach the credentials form regardless of an existing session in that tab.

### Session storage & transport
- **D-04:** Move the per-tab session token out of the shared httpOnly cookie into a tab-scoped store (`sessionStorage`, not `localStorage` — must not leak across tabs) attached via an `Authorization` header (or equivalent explicit per-request attachment) rather than relying on automatic cookie transport.
- **D-05:** Security trade-off explicitly accepted: since the token now lives in JS-readable storage, XSS blast radius is larger than with httpOnly cookies. Mitigate with short-lived tokens + a refresh mechanism, not by skipping the trade-off. This is judged acceptable given TRT PM is an internal tool with a known, small user base — not a public-facing consumer app.
- **D-06:** Every one of the ~26 call sites currently depending on `verifySession()` (`lib/dal.ts`) reading the shared cookie must be updated to also accept/prefer the per-tab token path for tabs using the new independent-session mechanism. Server Components, Server Actions, and Route Handlers all need a consistent way to read "whichever session this specific tab is using" — Claude's discretion on the exact mechanism (e.g. a small adapter in the DAL that checks for an incoming Authorization header first, falls back to the shared cookie).

### Sign-out semantics
- **D-07:** Signing out in one tab signs out only that tab's session. Other tabs — including other tabs logged in as the *same* user via their own independent per-tab session — are unaffected and continue working until they separately sign out or their own token expires. Do NOT implement cross-tab sign-out propagation/revocation for this phase.

### Claude's Discretion
- Exact refresh-token mechanism and token lifetime for the per-tab tokens (D-05) — pick something reasonable for an internal tool (e.g. 15–30 min access token + silent refresh), document the choice in the plan.
- Exact UI placement/wording of the "Sign in as a different user" entry point (D-02) — dashboard header vs. a link on the sign-in page vs. both.
- Internal mechanism for the DAL to distinguish "shared cookie session" vs "per-tab token session" (D-06) — as long as every existing `verifySession()` call site keeps working unchanged for the shared-cookie (default) case.

</decisions>

### Root-cause investigation
- `.planning/debug/auth-single-session-multi-tab.md` — full root-cause investigation (2026-07-17): confirms cookie-based JWT sessions are one-per-browser-origin by HTTP spec, not a server-side merge/corruption bug; identifies `app/(auth)/sign-in/page.tsx`'s unconditional `redirect('/dashboard')` on any existing session as the direct source of the reported "shows previous user" symptom; live-tested via cookie-jar reproduction against `npm run dev`.

### Existing auth architecture
- `auth.ts`, `auth.config.ts` — NextAuth v5 core: JWT session strategy, Credentials provider, `jwt()`/`session()` callbacks that attach `role`.
- `lib/dal.ts` — `verifySession()` (wrapped in `React.cache()`), the single authoritative session-read choke point used by ~26 files across Server Components/Actions/Route Handlers. Any new per-tab token path must integrate here, not bypass it.
- `actions/auth.ts` — sign-in/sign-out Server Actions; `signOut({ redirect: false })` + manual cookie clearing (already has a comment describing a related suspected bug class around stale tokens after switching test accounts).
- `proxy.ts` — Next 16's middleware-rename; does optimistic redirects based on session cookie presence only (authoritative checks stay in the DAL per this repo's established pattern — do not put per-tab-token logic here beyond an optimistic check).
- `app/(auth)/sign-in/page.tsx` — currently `if (session?.user) redirect('/dashboard')`; this exact check needs a carve-out for the new "sign in as different user" entry point (D-03).

No external specs/ADRs beyond the debug session file — this is a bug-driven architecture phase, not spec-driven.

<code_context>
## Existing Code Insights

### Reusable Assets
- `lib/dal.ts`'s `verifySession()` — the one choke point to extend, not replace. Every consumer already goes through here, so a per-tab-aware DAL change is the minimal-surface-area path to correctness.
- `actions/auth.ts`'s existing sign-out cookie-clearing logic — already handles clearing every known cookie-name variant + chunk suffixes; the reasoning there ("a browser that later signs in as a different test account can end up with a stale/ambiguous token") is directly relevant prior art for the new per-tab sign-out (D-07).

### Established Patterns
- Authoritative checks always happen in the DAL (`lib/dal.ts`), never trusted from `proxy.ts` alone — `proxy.ts` only does optimistic redirects. The per-tab session mechanism must follow this same split (optimistic check in proxy, authoritative check in DAL).

### Integration Points
- `app/(auth)/sign-in/page.tsx` (entry point for D-02/D-03)
- `lib/dal.ts::verifySession()` (integration point for D-04/D-06)
- `actions/auth.ts` (sign-in and sign-out Server Actions, for D-07)

</code_context>
