## User Constraints (from CONTEXT.md)

### Locked Decisions

**Trigger / UX model**
- **D-01:** A new tab opened normally (typed URL, clicked link, duplicate tab) still inherits the current shared session — no change to today's convenience for the common single-user case.
- **D-02:** A specific, explicit action — e.g. a "Sign in as a different user" link/button reachable from the sign-in page and/or dashboard — starts a fresh, independent per-tab session scoped to that tab only, without touching any other open tab's session.
- **D-03:** The current sign-in page behavior of unconditionally `redirect('/dashboard')` when *any* session exists must change for the "sign in as different user" entry point specifically — that path must always reach the credentials form regardless of an existing session in that tab.

**Session storage & transport**
- **D-04:** Move the per-tab session token out of the shared httpOnly cookie into a tab-scoped store (`sessionStorage`, not `localStorage` — must not leak across tabs) attached via an `Authorization` header (or equivalent explicit per-request attachment) rather than relying on automatic cookie transport.
- **D-05:** Security trade-off explicitly accepted: since the token now lives in JS-readable storage, XSS blast radius is larger than with httpOnly cookies. Mitigate with short-lived tokens + a refresh mechanism, not by skipping the trade-off. This is judged acceptable given TRT PM is an internal tool with a known, small user base — not a public-facing consumer app.
- **D-06:** Every one of the ~26 call sites currently depending on `verifySession()` (`lib/dal.ts`) reading the shared cookie must be updated to also accept/prefer the per-tab token path for tabs using the new independent-session mechanism. Server Components, Server Actions, and Route Handlers all need a consistent way to read "whichever session this specific tab is using" — Claude's discretion on the exact mechanism (e.g. a small adapter in the DAL that checks for an incoming Authorization header first, falls back to the shared cookie).

**Sign-out semantics**
- **D-07:** Signing out in one tab signs out only that tab's session. Other tabs — including other tabs logged in as the *same* user via their own independent per-tab session — are unaffected and continue working until they separately sign out or their own token expires. Do NOT implement cross-tab sign-out propagation/revocation for this phase.

- Exact UI placement/wording of the "Sign in as a different user" entry point (D-02) — dashboard header vs. a link on the sign-in page vs. both.
- Internal mechanism for the DAL to distinguish "shared cookie session" vs "per-tab token session" (D-06) — as long as every existing `verifySession()` call site keeps working unchanged for the shared-cookie (default) case.

## Summary

The existing shared-cookie session (NextAuth v5, JWT strategy, `authjs.session-token`) cannot be duplicated per-tab because cookies are inherently one-per-browser-origin — this was already conclusively established in `.planning/debug/auth-single-session-multi-tab.md`. This research confirms the phase is buildable **without any new npm dependency**: Auth.js's own low-level JWT primitives (`encode`/`decode`/`getToken`, re-exported from `@auth/core/jwt` via the already-installed `next-auth/jwt` subpath) can mint and verify a second, independently-transported token using the exact same encryption scheme (`AUTH_SECRET`-derived A256CBC-HS512 JWE) as the main session cookie — so there is no need to hand-roll JWT signing.

The critical, verified finding that shapes the whole plan: **Auth.js's built-in `getToken()` checks the cookie before the `Authorization` header** (confirmed by reading `node_modules/@auth/core/src/jwt.ts`). Since the browser will keep auto-attaching the *original* tab's shared cookie to every request the *new* per-tab-token tab makes (same browser, same origin, cookies are always sent automatically regardless of what JS does), a DAL adapter that reuses `getToken()`'s default precedence would silently resolve to the wrong (stale/other-tab) user. The DAL must invert this precedence explicitly: **check for an incoming `Authorization` header first, and only fall back to the shared cookie if absent** — which is exactly what D-06 in CONTEXT.md already specifies, and this research confirms *why* that ordering is load-bearing, not arbitrary.

The second critical finding: Next.js Server Actions invoked via a direct function call or `useActionState` **cannot have a custom header manually attached** to their underlying network request — there is no public API for this (confirmed via Next.js GitHub discussion #71801 and #74255). The per-tab token must instead travel as an **explicit bound/serialized argument** to the Server Action (e.g. `action.bind(null, token)` or a hidden form field) for the Server Action call path, while Route Handlers and client-triggered `fetch()` calls (including Next's own soft-navigation RSC fetches, which go through the global `fetch()`) **can** carry a real `Authorization: Bearer <token>` header via a client-side global-`fetch` wrapper — a well-documented (if unofficial) community pattern.

**Primary recommendation:** Extend `lib/dal.ts`'s `verifySession()` with a header-first, cookie-fallback adapter built on `next-auth/jwt`'s `decode()` (same secret/salt as the main session, so it's the same trusted issuer); mint per-tab access+refresh tokens with `encode()` from the same module; transport the access token via a global-`fetch` header-injection wrapper for Route Handlers/RSC navigations, and via an explicit bound argument for Server Actions; keep the default shared-cookie path (`auth()`, `signIn()`, `signOut()`) completely untouched.

### Pattern 1: Reuse Auth.js's own JWE primitives for a second, independently-transported token
**What:** Import `encode`/`decode` from `next-auth/jwt` (NOT `@auth/core/jwt` directly — `next-auth` is the direct dependency in `package.json`; `@auth/core` is transitive). Use the same `secret: process.env.AUTH_SECRET` and a `salt` value (Auth.js defaults `salt` to the cookie name, e.g. `"authjs.session-token"` — for the per-tab token, use a **distinct** salt string, e.g. `"trt-pm.tab-session"`, so a per-tab token can never be replayed into the shared-cookie slot or vice versa, even though it shares the same underlying secret).
**When to use:** Minting/verifying the per-tab access and refresh tokens.
**Example:**
```typescript
// Source: node_modules/@auth/core/src/jwt.ts (re-exported by next-auth/jwt) — verified by direct read
import { encode, decode } from "next-auth/jwt"

const TAB_SALT = "trt-pm.tab-session" // distinct from the default cookie-name salt

export async function mintTabAccessToken(userId: string, role: string) {
  return encode({
    secret: process.env.AUTH_SECRET!,
    salt: TAB_SALT,
    maxAge: 20 * 60, // 20 min — Claude's-discretion value, see Open Questions
    token: { sub: userId, role, typ: "access" },
  })
}

export async function verifyTabToken(token: string) {
  return decode({ secret: process.env.AUTH_SECRET!, salt: TAB_SALT, token })
  // returns the payload (or null if invalid/expired) — same guarantees as the main session JWT
}
```

### Pattern 2: DAL header-first, cookie-fallback adapter (implements D-06)
**What:** `verifySession()` must NOT use Auth.js's own `getToken()` convenience function as-is, because `getToken()` checks the cookie *before* the header (verified in source) — the opposite of what's needed here, since the shared cookie is always present in the browser regardless of which tab is "active."
**When to use:** Every one of the ~58 existing `verifySession()`/`requireRole()`/`requireAdmin()`/`requireOwnerOrAdmin()` call sites (grep-counted directly in this codebase — CONTEXT.md's "~26" figure appears to undercount; the real number spans `app/(app)/**/page.tsx`, `app/api/**/route.ts`, and `actions/*.ts`).
**Example:**
```typescript
// lib/dal.ts — extends the existing pattern, does not replace auth()
import "server-only"
import { cache } from "react"
import { headers } from "next/headers"
import { redirect, forbidden } from "next/navigation"
import { auth } from "@/auth"
import { verifyTabToken } from "@/lib/tab-session"
import { isAdminRole, type UserRole } from "@/lib/workflow"

export const verifySession = cache(async (): Promise<{ userId: string; role: UserRole }> => {
  // 1. Header first (per-tab token) — reversed from Auth.js's own getToken() precedence
  const incomingAuth = (await headers()).get("authorization")
  if (incomingAuth?.startsWith("Bearer ")) {
    const payload = await verifyTabToken(incomingAuth.slice(7))
    if (payload?.sub && payload.typ === "access") {
      return { userId: payload.sub as string, role: payload.role as UserRole }
    }
    // invalid/expired per-tab token: DO NOT silently fall back to the shared cookie —
    // that would resolve to the WRONG (other tab's) user. Redirect to sign-in instead.
    redirect("/sign-in")
  }
  // 2. Fall back to the existing shared-cookie path — UNCHANGED
  const session = await auth()
  if (!session?.user?.id) redirect("/sign-in")
  return { userId: session.user.id, role: session.user.role as UserRole }
})
```
**Route Handlers** need the same logic but reading `req.headers.get('authorization')` from the handler's own `Request`/`NextRequest` parameter instead of `next/headers` — both resolve to the same incoming header, so a shared helper function (not duplicated per call site) should back both `headers()`-based and `req`-based callers.

### Pattern 3: Server Action token transport — bound argument, NOT a header
**What:** Confirmed via Next.js GitHub discussions #71801 and #74255 — there is **no supported way** to attach a custom header to the network request Next.js's client runtime makes when invoking a Server Action (whether via `<form action={fn}>` or a direct call in a transition). The token must travel as an explicit argument instead.
**When to use:** Any Server Action a per-tab-session tab needs to call.
**Example:**
```typescript
// Client Component
'use client'
const tabToken = sessionStorage.getItem('tabAccessToken')
const boundAction = someServerAction.bind(null, tabToken) // token becomes a real argument, serialized like any other

// Server Action
'use server'
export async function someServerAction(tabToken: string | null, formData: FormData) {
  const session = tabToken
    ? await verifyTabTokenOrRedirect(tabToken)
    : await verifySession() // existing shared-cookie path
  // ...
}
```
For `useActionState`-bound forms specifically, the same `.bind(null, tabToken)` pattern applies before passing the action into `useActionState`.

### Pattern 4: Global `fetch` wrapper for Route Handlers + client-side RSC navigation
**What:** `[CITED: community pattern, verified via GitHub discussions #74255 and #58110 — not officially documented by the Next.js team, MEDIUM confidence]`. Next.js's client router uses the browser's global `fetch()` under the hood both for explicit `fetch()` calls your code makes (e.g. to `/api/*` Route Handlers) AND for its own soft-navigation RSC payload fetches (triggered by `<Link>`/`router.push`). Overriding `window.fetch` once, early, in a mounted client component lets a single wrapper inject the header for both cases.
**When to use:** Mount once, near the root layout, gated so it only activates for tabs that actually have a `sessionStorage` per-tab token (default single-session tabs must be completely unaffected — untouched fetch behavior).
**Example:**
```typescript
'use client'
// app/_components/tab-session-provider.tsx
useEffect(() => {
  const token = sessionStorage.getItem('tabAccessToken')
  if (!token) return // default shared-cookie tab: do not touch fetch at all
  const originalFetch = window.fetch
  window.fetch = (input, init = {}) => {
    const headers = new Headers(init.headers)
    headers.set('Authorization', `Bearer ${sessionStorage.getItem('tabAccessToken')}`)
    return originalFetch(input, { ...init, headers })
  }
  return () => { window.fetch = originalFetch }
}, [])
```
**Important caveat surfaced by this research:** this wrapper CANNOT help the very first hard-navigation page load of a new tab (typed URL into an address bar, or a plain `<a href>` navigation) — a real browser document GET/POST happens before any JS in that tab has run, so no header can be attached. This is not a gap in practice for D-02's flow specifically, because "sign in as a different user" is a same-page (already-loaded) Server Action call followed by a **client-triggered** `router.push()` (soft navigation, which the wrapper DOES cover) — see Pattern 5. It would be a gap only if a user tried to bookmark/reload a per-tab-token-protected URL directly in a fresh tab, which is out of scope per D-01 (a normal new tab intentionally inherits the shared session, not a per-tab token).

### Pattern 5: The "return tokens from the Server Action, then client `router.push`" sign-in flow (closes the token hand-off gap)
**What:** The default `signinAction` uses `next-auth`'s own `signIn(..., { redirectTo: '/dashboard' })`, which does a **server-side** redirect — the client never has a chance to receive/store anything before navigation happens. The new per-tab sign-in action must do the opposite: verify credentials, mint tokens, and **return** them in its `useActionState` result (no `redirect()` call inside the action). The calling Client Component then writes the tokens to `sessionStorage` and calls `router.push('/dashboard')` itself — a genuine client-side soft navigation that Pattern 4's fetch wrapper will correctly intercept (the token is already in `sessionStorage` before this navigation's RSC fetch fires).
**Why this matters:** Without this sequencing, the freshly-minted per-tab token would exist only in the server-action's return value with nowhere for the very next page load to read it from, since Server Components render before client-side `useEffect`s run.
**Example:**
```typescript
// actions/tab-auth.ts — 'use server'
export async function tabSigninAction(_prev: TabSigninState, formData: FormData) {
  // validate + verify credentials — REUSE the same lookup/bcrypt logic as auth.ts's
  // Credentials provider `authorize()` (extract into a shared `verifyCredentials()`
  // helper — do not duplicate password-verification logic, see Don't Hand-Roll)
  const user = await verifyCredentials(email, password)
  if (!user) return { message: 'Invalid email or password.' }
  const accessToken = await mintTabAccessToken(user.id, user.role)
  const refreshToken = await mintTabRefreshToken(user.id)
  return { accessToken, refreshToken, expiresAt: Date.now() + 20 * 60 * 1000 }
  // NO redirect() here — the client does it after storing tokens
}
```
```typescript
// new-session-form.tsx — 'use client'
const [state, formAction] = useActionState(tabSigninAction, {})
useEffect(() => {
  if (state.accessToken) {
    sessionStorage.setItem('tabAccessToken', state.accessToken)
    sessionStorage.setItem('tabRefreshToken', state.refreshToken)
    sessionStorage.setItem('tabTokenExpiresAt', String(state.expiresAt))
    router.push('/dashboard')
  }
}, [state])
```

### Pattern 6: `proxy.ts` optimistic gate — presence-only header check (D-06's "authoritative checks stay in the DAL" boundary)
**What:** `auth.config.ts`'s existing `authorized({ auth, request })` callback (consumed by `proxy.ts`'s `NextAuth(authConfig).auth` default export) currently only checks `!!auth?.user` (cookie-derived). Extend it to also treat a present `Authorization` header as sufficient for the optimistic pass-through — **presence only, no decode**, matching the existing "Proxy should not do slow/heavy checks" convention documented in Next.js's own bundled docs.
**Example:**
```typescript
// auth.config.ts
authorized({ auth, request }) {
  const isLoggedIn = !!auth?.user || request.headers.has('authorization')
  const path = request.nextUrl.pathname
  const isPublic = [...].some(...)
  if (isPublic) return true
  return isLoggedIn
}
```
**Why needed at all:** in the common case the shared cookie already exists in the browser (per D-01, a session usually already exists before someone opens a second per-tab identity), so `auth?.user` alone would already pass proxy's gate even for a per-tab tab. This patch only matters for the edge case of a genuinely fresh browser (no cookie at all yet) whose *first* authenticated identity is established via the per-tab mechanism — without it, `router.push('/dashboard')` from Pattern 5 would get redirected back to `/sign-in` by proxy before the DAL ever gets a chance to read the header.

### Anti-Patterns to Avoid
- **Calling `next-auth`'s `signIn()`/`signOut()` for the per-tab flow:** These functions manage the shared cookie directly — calling them from the new "sign in as different user" path would overwrite the *other* tabs' session, reproducing the exact bug this phase fixes. The per-tab path must verify credentials independently (via a shared `verifyCredentials()` helper) and mint tokens manually.
- **Using Auth.js's `getToken()` unmodified in the DAL:** Its cookie-before-header precedence is backwards for this use case (see Pattern 2). Only use `decode()` directly with explicit header-first logic.
- **Storing the per-tab token in `localStorage`:** Explicitly ruled out by D-04 — `localStorage` is shared across all tabs of the same origin, defeating the entire purpose.
- **Letting an invalid/expired per-tab token silently fall back to the shared cookie:** Would resolve to the *wrong* user (whoever is logged in via the cookie in a different tab), not "logged out." Fail closed (redirect to sign-in) instead, as shown in Pattern 2.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| JWT signing/encryption for the per-tab token | A second `jose`-based sign/verify implementation | `encode()`/`decode()` from `next-auth/jwt` | Same encryption scheme (A256CBC-HS512 JWE) Auth.js already uses for the main session — zero new crypto code, one secret to manage (`AUTH_SECRET`), already battle-tested by the installed dependency |
| Credential verification for the per-tab sign-in path | A copy-pasted bcrypt-compare block inside the new Server Action | Extract the existing Credentials provider's `authorize()` logic (in `auth.ts`) into a shared `verifyCredentials(email, password)` helper, call it from both places | Avoids two divergent copies of password-verification logic — a classic source of an auth bypass bug when one copy gets a security fix and the other doesn't |
| Refresh-token rotation/replay-detection | A custom token-family/rotation-tracking table | Nothing this phase — stateless JWT with natural expiry, explicitly accepted per D-05/D-07 | Building real rotation-with-reuse-detection requires a DB-backed token store, which is out of scope (no cross-tab revocation this phase); document as a known limitation instead (see Open Questions) |

**Key insight:** The entire phase is achievable by *composing* primitives Auth.js already ships (`encode`/`decode`) with primitives the browser already ships (`sessionStorage`, `fetch`) — there is no auth-library gap here, only a request-context/precedence-ordering problem in the DAL and a header-transport problem for Server Actions specifically.

## Common Pitfalls

### Pitfall 1: Trusting `getToken()`'s default cookie-before-header precedence
**What goes wrong:** A per-tab-token tab's requests resolve to the *other* tab's user (from the still-present shared cookie), not the intended per-tab user — this is the exact "shows the previous user" bug from the original debug session, reintroduced at a different layer.
**Why it happens:** `getToken()` (verified in `node_modules/@auth/core/src/jwt.ts`) checks `sessionStore.value` (cookie) first and only inspects the `authorization` header if the cookie slot is empty. The shared cookie is essentially *always* present in a non-incognito browser.
**How to avoid:** Never call `getToken()` directly for this phase's adapter — build the header-first check manually (Pattern 2), or write a custom wrapper that reorders header/cookie precedence explicitly.
**Warning signs:** Manual QA showing "signed in as User B in tab B, but a Server Component in tab B renders User A's name" — this is the signature of the precedence bug, not a token-minting bug.

### Pitfall 2: Assuming a Server Action can read a client-attached custom header
**What goes wrong:** Code written as if `headers().get('authorization')` inside a Server Action will reflect a header the client explicitly set on that action call — it won't, because there is no client-side hook to set it (see Pattern 3).
**Why it happens:** It *does* work for genuine `fetch()` calls to Route Handlers and for Next's own RSC soft-navigation requests (both go through `window.fetch`), which makes it easy to wrongly assume Server Actions behave the same way — they use a different internal transport that the app cannot instrument.
**How to avoid:** Always pass the per-tab token as an explicit bound/serialized argument to Server Actions (Pattern 3), never rely on the header for that call path.
**Warning signs:** A Server Action's `headers()` call returning the header from a *previous, unrelated* navigation request rather than the Server Action's own invocation (a known source of confusion in community reports).

### Pitfall 3: Minting/storing the per-tab token before the client can capture it
**What goes wrong:** If the new "sign in as different user" action does a `redirect()` server-side (mirroring the existing `signinAction`), the freshly-minted token is discarded — the very next page load has nowhere to read it from, since Server Components render before any client `useEffect` runs.
**Why it happens:** Copy-pasting the existing `signinAction`'s `redirectTo` pattern is the natural first instinct, but that pattern is specifically designed around the shared cookie being set server-side (no client hand-off needed) — the per-tab flow has the opposite requirement.
**How to avoid:** The per-tab sign-in action must `return` the tokens (not `redirect()`), and the calling Client Component must write to `sessionStorage` and *then* call `router.push()` itself (Pattern 5).

### Pitfall 4: `proxy.ts`/`authorized()` blocking a genuinely-fresh browser's first per-tab identity
**What goes wrong:** In a browser with no shared cookie at all yet, `router.push('/dashboard')` after per-tab sign-in gets redirected back to `/sign-in` by the optimistic proxy gate, even though the per-tab token is valid — because `authorized()` only checks `auth?.user` (cookie-derived).
**Why it happens:** The proxy layer was written assuming the only signal of "logged in" is the cookie.
**How to avoid:** Extend `authorized()` per Pattern 6 (presence-only header check, no decode).
**Warning signs:** Works in manual QA (where a cookie usually already exists from an earlier session) but fails in a genuinely clean browser profile / fresh incognito-adjacent test.

### Pitfall 5: The existing `signoutAction`'s multi-cookie-name clearing logic does not apply to per-tab sign-out
**What goes wrong:** Reusing `actions/auth.ts`'s `signoutAction` (which clears every `authjs.session-token`/`__Secure-`/chunked cookie variant) for the per-tab "sign out this tab" action would incorrectly nuke the *shared* cookie too — signing every other tab out, violating D-07.
**Why it happens:** It's the only existing sign-out code in the repo, so it's a tempting reuse target.
**How to avoid:** Per-tab sign-out is purely client-side: clear this tab's three `sessionStorage` keys (`tabAccessToken`, `tabRefreshToken`, `tabTokenExpiresAt`) and redirect within the tab. No server round-trip is strictly required (though a server-side "log this event" call is fine); definitely do not call the existing `signoutAction`/`next-auth`'s `signOut()` from this path.

## Code Examples

### Reading the incoming Authorization header identically from a Server Component/Action vs. a Route Handler
```typescript
// Source: node_modules/next/dist/docs/01-app/02-guides/authentication.md (DAL pattern) +
// node_modules/@auth/core/src/jwt.ts (header-reading precedent) — composed for this phase

// lib/tab-session.ts
import "server-only"
import { headers } from "next/headers"
import { decode } from "next-auth/jwt"

const TAB_SALT = "trt-pm.tab-session"

// Works from Server Components AND Server Actions (both have next/headers access)
export async function readIncomingTabToken(): Promise<string | null> {
  const h = await headers()
  const auth = h.get("authorization")
  return auth?.startsWith("Bearer ") ? auth.slice(7) : null
}

// Route Handlers pass their own Request explicitly (same underlying header)
export function readTabTokenFromRequest(req: Request): string | null {
  const auth = req.headers.get("authorization")
  return auth?.startsWith("Bearer ") ? auth.slice(7) : null
}

export async function verifyTabToken(token: string) {
  try {
    return await decode({ secret: process.env.AUTH_SECRET!, salt: TAB_SALT, token })
  } catch {
    return null
  }
}
```

## Open Questions

1. **Exact access/refresh token lifetimes**
   - What we know: D-05 asks for "short-lived + refresh," CONTEXT.md's discretion note suggests "15–30 min access + silent refresh" as a reasonable ballpark.
   - What's unclear: No hard requirement from the user beyond "reasonable for an internal tool."
   - Recommendation: 20 min access / 8 hour refresh (roughly a working day), both minted via `encode()` with a `typ` claim (`"access"` vs `"refresh"`) so one can never be used as the other. Document this choice explicitly in the plan per CONTEXT.md's instruction.

2. **Refresh token storage location**
   - What we know: D-05 accepts `sessionStorage` exposure for the access token; asks whether the refresh token needs different treatment "to avoid making refresh tokens as exposed as access tokens."
   - What's unclear: A per-tab-scoped httpOnly cookie is not possible (cookies aren't tab-scoped, which is the entire problem this phase solves) — there is no browser storage primitive that is both tab-scoped AND JS-inaccessible.
   - Recommendation: Accept the same trade-off for the refresh token as the access token (both in `sessionStorage`) — this is the only mechanism that satisfies "tab-scoped." Mitigate purely via a shorter refresh-token lifetime than a typical OAuth refresh token (hours, not weeks) rather than trying to hide it from JS, which isn't achievable per-tab. Flag this explicitly to the user in the plan as the accepted consequence of D-04's tab-scoping requirement.

3. **Does Pitfall/Assumption A1 hold in this exact Next 16.2.9 + React 19.2.4 setup?**
   - What we know: Confirmed as a working pattern in community reports against various Next 15.x App Router versions; Next 16's proxy/RSC architecture is described as functionally unchanged from 15's middleware in the bundled docs.
   - What's unclear: No direct confirmation against Next 16.2.9 specifically.
   - Recommendation: First task in the plan should be a minimal smoke test (mount the fetch wrapper on a throwaway route, verify the header appears in the Network tab on a `router.push` soft nav) before committing the rest of the plan's tasks to this mechanism.

### Applicable ASVS Categories

| ASVS Category | Applies | Standard Control |
|---------------|---------|-----------------|
| V2 Authentication | yes | Credential verification reuses the existing bcrypt-based `authorize()` logic (extracted into a shared helper, Don't Hand-Roll) — no new authentication mechanism, just a second transport for an already-verified identity |
| V3 Session Management | yes | Per-tab tokens are short-lived, signed/encrypted JWEs (`next-auth/jwt` `encode`/`decode`, same AUTH_SECRET-derived key as the main session) — not opaque unsigned tokens; explicit `typ` claim prevents refresh-token-as-access-token confusion |

| V5 Input Validation | yes | New per-tab sign-in Server Action validates email/password with `zod` (same pattern as existing `SigninSchema`) |
| V6 Cryptography | yes | **Do not hand-roll** — reuse `next-auth/jwt`'s `encode`/`decode` (A256CBC-HS512 JWE, already the app's established scheme) rather than introducing a second crypto library/scheme |

### Known Threat Patterns for this stack

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| XSS reading the per-tab token out of `sessionStorage` | Information Disclosure | **Explicitly accepted trade-off per D-05** (internal tool, small known user base) — mitigated by short access-token lifetime + no ability to read the refresh token's claims client-side (opaque JWE ciphertext, not a plain JWT) |
| A stale/expired per-tab token being silently treated as "log back into the shared cookie identity" | Spoofing | Pitfall/Anti-pattern noted above: fail closed (redirect to sign-in) on invalid per-tab token, never silently fall back to the cookie once a header was present — falling back would let an attacker with a stolen-but-expired token effectively "downgrade" a victim's tab to whatever the shared cookie currently holds |
| CSRF against the new per-tab sign-in/refresh Server Action or Route Handler | Tampering | Server Actions in Next.js already carry Next's built-in Server Action CSRF protections (Origin header check); the per-tab path is **less** CSRF-exposed than the cookie path in one sense — bearer tokens in a header are never auto-attached cross-site the way cookies are, so a forged cross-site request cannot carry a valid per-tab `Authorization` header at all |
| Token replay across the two token "slots" (using a per-tab token where a shared-cookie JWT was expected, or vice versa) | Tampering / Elevation of Privilege | Distinct `salt` value for `encode`/`decode` (Pattern 1, Assumption A3) plus an explicit `typ` claim (`"access"`/`"refresh"`) makes cross-slot replay fail decode/type-check rather than silently succeeding |
