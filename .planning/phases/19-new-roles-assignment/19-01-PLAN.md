<objective>
Convert `users.position` from nullable free `text()` to a DB-enforced Postgres enum (`ROLE-04`), establishing a single `POSITION_VALUES` source of truth in `lib/workflow.ts` that both the Drizzle `pgEnum` and every position-picking UI (profile select, configurator) consume. This is the single highest-risk task in Phase 19 — a type CONVERSION on a live column that already holds non-null values for real users — so it is isolated in its own Wave-1 plan, inspected before it runs, and gated by a blocking human checkpoint that mirrors the Phase 17 migration rigor.

Purpose: `requiredPosition` gating (already shipped ad hoc against free-text `position`) becomes DB-enforced; downstream Phase 19 plans (dashboards in 19-02, profile/configurator in 19-03) build on the enum type this plan creates. PAY-02's future narrowing (Phase 20) also depends on this.
Output: `positionEnum` in the DB, `users.position` retyped, `POSITION_VALUES`/`POSITION_LABELS` exported from `lib/workflow.ts`, an inspection script, and an idempotent migration script.
</objective>

<decision>
D-19-01-A (DB-enforced target of ROLE-04): ONLY `users.position` converts to the Postgres enum — that is the exact clause ROLE-04's success criterion enforces ("users.position converts from free text to a DB-enforced Postgres enum"). `workflow_step_definitions.requiredPosition`, `receiverRequiredPosition` stay `text()` but are sourced exclusively from an enum-constrained `<select>` in the Configurator (built in 19-03), giving app-level validation without the added migration surface of converting three live columns. The gate (`actingUser.position !== requiredPos`) is a string comparison and works identically whether `requiredPosition` is enum-typed or free text holding an enum value.

D-19-01-B (value set is data-derived, not guessed): The enum value set is the UNION of (1) the three known machine gating values `head_of_operations`, `head_designer`, `chief_production_officer`, and (2) EVERY distinct non-null value currently stored in `users.position` / `requiredPosition` / `receiverRequiredPosition`, added VERBATIM so no super-admin title display value (ROLE-03: "Head of Projects", "MD", etc.) is lost. Only demonstrably junk/placeholder values (e.g. the literal profile placeholder string) are backfilled to null, and only with explicit approval at the Task 3 checkpoint. This reconciles the doc wording "ops_factory" vs the live enum's already-shipped `factory_operations` at the requirement layer, not by renaming anything.
</decision>
