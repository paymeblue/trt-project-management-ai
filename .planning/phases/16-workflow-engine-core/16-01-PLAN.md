No engine logic yet — schema only, then push it to the database.

  <action>
    Add to db/schema.ts, following the file's existing style (snake_case DB names, `uuid('...').primaryKey().defaultRandom()`, `timestamp('...').defaultNow().notNull()`):

    1. New enum `fulfillmentKindEnum = pgEnum('fulfillment_kind', ['creation','checklist','readiness','ack','yes_no_upload','approval','assignment'])` — the 4 existing StepKind values plus the 3 new ones from WF-03.

    2. New table `workflowStepDefinitions` → DB name `workflow_step_definitions`, columns: `id` uuid PK defaultRandom; `graph` text notNull default 'live' (isolates the Phase 16 test graph from the live graph); `stepKey` text('step_key') notNull; `label` text notNull; `role` roleEnum('role') notNull; `fulfillmentKind` fulfillmentKindEnum('fulfillment_kind') notNull; `checklistSlug` text('checklist_slug') (nullable — set only when fulfillmentKind = 'checklist', mirrors checklist_definitions.slug); `targetRole` roleEnum('target_role') (nullable — the role an 'assignment' step lets the actor pick a user from); `isOptional` boolean('is_optional') notNull default false; `orderIndex` integer('order_index') notNull (display/default ordering only — adjacency lives in edges); `createdAt`; `updatedAt`. Add composite unique on (graph, stepKey) via `unique().on(t.graph, t.stepKey)`.

    3. New table `workflowStepEdges` → DB name `workflow_step_edges`, columns: `id` uuid PK; `graph` text notNull default 'live'; `fromStepId` uuid('from_step_id') notNull references workflowStepDefinitions.id onDelete cascade; `toStepId` uuid('to_step_id') notNull references workflowStepDefinitions.id onDelete cascade; `createdAt`. Add composite unique on (fromStepId, toStepId). These edges are the ONLY source of adjacency — a join step is one with multiple incoming edges (WF-05).

    4. New table `workflowStepStates` → DB name `workflow_step_states`, per-project runtime state for the new kinds: `id` uuid PK; `projectId` uuid('project_id') notNull references projects.id onDelete cascade; `stepDefId` uuid('step_def_id') notNull references workflowStepDefinitions.id onDelete cascade; `status` text notNull default 'pending' (values 'pending' | 'sent' | 'complete'); `answer` text (nullable — 'yes' | 'no' for yes_no_upload); `uploadData` text('upload_data') (nullable — base64 data URL, same pattern as checklists.photoData); `uploadName` text('upload_name') (nullable); `assignedUserId` uuid('assigned_user_id') references users.id (nullable — for assignment); `sentBy` uuid('sent_by') references users.id (nullable — approval send); `receivedBy` uuid('received_by') references users.id (nullable — approval receive); `actedBy` uuid('acted_by') references users.id (nullable); `updatedAt`. Add composite unique on (projectId, stepDefId).

    5. Extend the EXISTING `projectStepCompletions` table (do not create a new one): add `stepDefId` uuid('step_def_id') references workflowStepDefinitions.id onDelete cascade (nullable — legacy live rows key by stepN/stepKey, graph-engine rows key by stepDefId); add `graph` text notNull default 'live'; add `skipped` boolean notNull default false (so a skipped optional step is recorded as a satisfied predecessor for join readiness, distinct from a genuine completion). Keep stepKey/stepN as-is for backward compatibility — do NOT remove them.

    Do NOT modify lib/workflow.ts, actions, or any consumer in this task. Schema only.
