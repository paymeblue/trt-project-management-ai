# Summary — 260719-90m: jsdom provider tests

Closes the acknowledged gap from 260718-u7d: the provider's timer/override
behavior was proven only by live browser runs (node-only test env). Added
`jsdom@^29` as a devDependency with a per-file `@vitest-environment jsdom`
pragma (global env stays `node`), widened the vitest include to
`tests/**/*.test.{ts,tsx}`, and wrote
`tests/app/tab-session-provider.test.tsx` — 8 cases rendered via
`react-dom/client` + React 19 `act`, with a mocked `window.location`
(jsdom can't navigate) and fake timers:

1. Token-less tab is a strict no-op (fetch identity untouched, no redirect).
2. Token at mount on a normal path → override installed + native bounce to
   `/tab-session/restore?to=<path+search>`.
3. Token at mount ON the restore route → override installed, no bounce
   (loop guard).
4. Override injects `Authorization` from the CURRENT sessionStorage token at
   call time (survives a same-tab user switch).
5. Silent refresh reads the refresh token at FIRE time — a pending timer can
   never restore the previous user's session (regression for aa3de81).
6. Activate event on an active tab re-arms the timer against the NEW
   session's expiry; the old timer provably never fires (regression for
   aa3de81's stomp fix).
7. Failed refresh clears all three session keys (D-20.1-03-B).
8. Unmount restores the original `window.fetch`.
