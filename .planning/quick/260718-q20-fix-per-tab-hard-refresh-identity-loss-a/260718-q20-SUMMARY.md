# Summary — 260718-q20

## 1. Hard-refresh per-tab identity loss (Phase 20.1 reopened) — FIXED, live-verified

**Why the three prior attempts failed:** all tried to correct the wrong-identity render
in-document. `router.refresh()`/same-path `replace()` bypassed the `window.fetch`
override server-side, and `push()` between two routes that both live inside the `(app)`
layout group reuses the client-cached layout segment — the identity chrome (sidebar/
header from `app/(app)/layout.tsx`) never re-mounts, and reconciliation against the
already-committed wrong-identity tree blocked the corrected paint.

**Fix (commit 88eb1a3):** native-navigation restore bounce.
- `TabSessionProvider` mount-time activation with a token now does
  `window.location.replace('/tab-session/restore?to=<current>')` — a NATIVE navigation
  to a new identity-agnostic client route OUTSIDE the `(app)` group.
- The fresh document starts with an EMPTY client Router Cache; after the provider's
  effect installs the fetch override (page defers one task via `setTimeout(0)` since
  child effects run before parent effects in the hydration commit), the restore page
  `router.replace(to)`s back. That RSC fetch carries the Bearer header and mounts the
  entire `(app)` tree — layout chrome included — as the per-tab user.
- `to` validated (internal path only, not the restore route itself); no-token visitors
  get a plain native redirect; `/tab-session/restore` added to `auth.config.ts` public
  paths so the flow survives a missing shared cookie.

**Live proof (dev server, agent-browser, QA accounts):**
- Shared cookie qa.factory + per-tab qa.ops2 on `/admin/dashboard`; hard reload →
  qa.ops2's full admin chrome repainted. `performance.getEntriesByType('navigation')`
  shows the document was natively loaded at `/tab-session/restore?to=%2Fadmin%2Fdashboard`
  — the bounce mechanism, not luck.
- New tab opened from the per-tab tab inherits sessionStorage (browser behavior) and
  correctly bounces to the per-tab identity too.
- Pure shared-cookie tab (no sessionStorage): zero bounce (navigation entry is the page
  itself), zero behavior change — strict no-op confirmed.

## 2. Per-tab notification mark-read scope hole — FIXED, live-verified

`markNotificationsReadAction` used bare `verifySession()`; Server Action POSTs carry no
Authorization header, so a per-tab tab's mark-read mutated the SHARED cookie user's
rows while the feed showed the per-tab user's. Migrated to the proven
`verifySessionForAction(tabToken)` bound-argument pattern (actions/profile.ts,
D-20.1-04-A); the bell passes `getTabToken()` at both call sites (commit 2199713).

**Live proof:** seeded 1 unread notification each for qa.ops2 and qa.factory; per-tab
qa.ops2 tab clicked "Mark all read" → DB shows qa.ops2 unread 2→0, qa.factory unread
still 1 (untouched). Seeded rows deleted afterwards.
