# Quick Task 260718-q20: hard-refresh identity restore + notification mark-read scope

## Problem 1 — hard-refresh loses per-tab identity (Phase 20.1 reopened)

A hard refresh (F5 / typed URL) is the browser's native top-level navigation — it runs
before any JS, so it can never carry the per-tab `Authorization` header. The server
resolves the shared cookie's user and the tab paints the WRONG identity.

Three prior in-document correction attempts failed:
- `59ef455` `router.refresh()` — server log showed the refresh request resolved via the
  shared cookie, not the header.
- `fd3ed2c` `router.push(scratch)` + `router.replace(current)` — server resolved the
  header identity correctly, but the corrected content never painted.
- `c7cd590` `router.push('/dashboard')` alone — same class of failure risk: `/dashboard`
  sits INSIDE the `(app)` layout group, and a soft navigation between two routes that
  share the `(app)` layout reuses the client-cached layout segment — the header/sidebar
  identity chrome (rendered by `app/(app)/layout.tsx`) is never re-mounted, and Router
  Cache reconciliation against the already-committed wrong-identity tree is exactly
  where the previous attempts died.

### Fix: native-navigation restore route (out-of-tree round trip)

1. `TabSessionProvider.activate({forceRerender})` (mount-time, token present, NOT already
   on the restore route) → `window.location.replace('/tab-session/restore?to=<current>')`.
   Native navigation → brand-new document → EMPTY client Router Cache, no committed
   wrong-identity tree to reconcile against.
2. New route `app/tab-session/restore/page.tsx` — client spinner page OUTSIDE the `(app)`
   layout group, identity-agnostic (renders the same for everyone, no data fetch).
3. On the restore page, after the provider's effect has installed the fetch override
   (child effects run before the parent's in the same hydration commit, so a
   `setTimeout(0)` in the page defers past the provider's install), `router.replace(to)`
   soft-navigates to the original URL. That RSC fetch carries the Bearer header, and the
   `(app)` layout is NOT part of the restore document's tree — the whole `(app)` segment
   mounts fresh under the per-tab identity: header, sidebar, page, everything.
4. `to` is validated: internal path only (must start `/`, not `//`, not `/tab-session`),
   else `/dashboard`. No-token visitors get a plain native redirect (cookie identity).
5. `auth.config.ts`: add `/tab-session/restore` to the public path list so the flow
   survives a missing/expired shared cookie (the page carries no data).

The existing event-driven activation path (`new-session-form.tsx` →
`TAB_SESSION_ACTIVATE_EVENT`) is live-verified working and is untouched (no
`forceRerender`).

## Problem 2 — notification mark-read acts as the shared-cookie user

`markNotificationsReadAction` uses bare `verifySession()`. Server Action POSTs carry no
Authorization header (architectural, D-20.1-04-A), so in a per-tab tab the bell FEED is
the per-tab user's (route handler, header-scoped) but MARK-READ mutates the shared-cookie
user's rows — cross-identity scope violation.

### Fix: bound-token pattern (proven in actions/profile.ts, Phase 20.1-04)

- `markNotificationsReadAction(tabToken: string | null, id?: string)` →
  `verifySessionForAction(tabToken)`.
- `notifications-bell.tsx` passes `getTabToken()` (lib/use-tab-token.ts) at both call
  sites.
