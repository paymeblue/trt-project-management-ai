# Quick Task 260718-qs5: complete the per-tab Server Action migration

## Problem

Phase 20.1 left a documented scope limitation (D-20.1-04-A): ~38 DAL call sites across
16 action files still used bare `verifySession()`/`requireAdmin()`. Server Action POSTs
carry no Authorization header, so any of those actions invoked from a per-tab session
tab silently acted as the SHARED cookie's user. Only profile + notifications had been
migrated to the proven bound-token pattern.

## Approach

1. `lib/dal.ts`: add `requireAdminForAction(tabToken)` sibling.
2. Every mutating action gains a REQUIRED `tabToken: string | null` first parameter →
   any missed or future call site is a compile error, not a silent identity bug.
3. Client call sites: `getTabToken()` for programmatic calls; `.bind(null, getTabToken())`
   for `useActionState`; new reusable `TabTokenForm` client wrapper for the 7 forms
   rendered from server components (issues x3, admin issues, disputes, about,
   email-formats).
4. `workflow-config.ts`: split the PIN-unlock check into an identity-parameterized
   `isUnlockedFor(userId)` core — actions validate the unlock cookie against the
   PER-TAB identity; the server-component path (`isConfiguratorUnlocked`) keeps
   header-aware `requireAdmin()`.
5. Exempt by design: `auth.ts`/`tab-auth.ts`/`email-auth.ts` (session establishment),
   `signOutAction` (shared-cookie sign-out), `confirmDualRoleStepAs` (auth-free CLI
   harness core).
6. Tests: dal mocks expose `verifySessionForAction`; call sites pass null; new
   token-forwarding assertion + 4 `requireAdminForAction` tests.

## Verification

- `tsc --noEmit` 0 errors, lint 0 errors, `npm test` 233 passed + 1 todo.
- Live cross-identity proof (dev server, real browser): shared cookie = qa.factory
  (NON-admin), per-tab = qa.ops2 (admin). About-page edit (TabTokenForm) saved and
  `static_content.updated_by` = qa.ops2 — impossible via the shared cookie identity,
  which is not admin. Reverted after.
- `requireAdminForAction` path: configurator PIN gate in the per-tab tab returned a
  clean "Incorrect PIN." on a wrong PIN (auth passed, PIN logic ran).
- Shared-cookie regression: cleared sessionStorage, rendered /factory-pm/dashboard,
  /factory-pm/product-readiness, /processes, /profile, /about — no bounce, no errors.
